# Chatbot 流式交互锁约束

## 背景

本项目使用 TDesign `Chatbot` 组件承载对话 UI，但消息流和交互编排不是纯组件默认行为。我们同时注入了：

- 自定义 SSE 协议转换
- 动态 `suggestion` / `activity-form` 内容
- 本地消息收尾逻辑
- 会话存储刷新与历史回填

因此，不能把“当前轮回复是否允许继续交互”只交给 TDesign 默认状态管理。

这类问题已经重复出现多次，后续改动必须遵守本文档。

## TDesign 行为结论

基于当前接入版本：

- `ChatSender.loading` 主要控制发送按钮/停止按钮状态。
- `ChatSender.disabled` 才是整体输入禁用。
- `ChatEngine` 的 `request:complete` 表示底层请求完成，不等于本项目的自定义 UI 注入和收尾已经完成。

结论：

- 不能只传 `loading`，必须同时传 `disabled`。
- 不能只监听 `request:start/request:complete` 做交互锁。

## 项目内唯一规则

### 1. 唯一交互锁

页面层必须维护一个统一交互锁，覆盖：

- 底部输入框发送
- 建议项点击发送
- 气泡内表单提交
- 以后新增的任何自定义消息交互入口

交互锁开启条件：

- 用户发起一轮新的 assistant 请求时立即开启
- 或底层引擎仍处于请求中

交互锁关闭条件：

- 只能在统一收尾函数执行完成后关闭
- `done`
- `error`
- `abort`

都必须进入同一个收尾出口。

### 2. 唯一发送入口

不要在各处直接裸调 `chatbotRef.sendUserMessage()`。

必须统一走包装发送函数，至少包含：

- 锁状态检查
- 重复发送 warning
- 发送前加锁
- 发送异常时解锁

如果新增入口绕过这个包装函数，默认视为 bug。

### 3. 唯一解锁入口

不要在事件监听里直接把页面交互锁设为 `false`。

`request:complete` 只能表示引擎层状态变化，不能直接解锁页面交互。页面交互解锁必须只发生在统一收尾函数里。

## 表单与建议项规则

对流式返回中的 `form` / `suggestion` 内容，规则如下：

- 可以提前渲染
- 在当前轮 assistant 回复真正完成前不可操作
- 只有收尾完成后才恢复交互

这条规则适用于：

- `activity-form`
- `suggestion`
- 后续新增的自定义 tool / activity 交互组件

如果组件在流式中展示出来但仍可点击、可选择、可提交，就是实现错误。

## 开发 Checklist

修改聊天区相关功能时，提交前必须检查：

1. 当前轮回复进行中时，输入框是否同时 `loading + disabled`。
2. 建议项点击是否走统一发送入口。
3. 表单提交是否走统一发送入口。
4. 新增的交互组件是否受统一交互锁控制。
5. 是否只有统一收尾函数负责解锁。
6. `error` / `abort` 路径是否也能解锁。
7. 测试是否覆盖“流未完成前不可操作，完成后恢复”。

## 推荐测试场景

- 纯文本流式回复期间，不能发送第二条消息。
- `form` chunk 已到达但未 `done` 时，表单字段和提交按钮不可用。
- `suggestion` chunk 已到达但未 `done` 时，点击只提示等待，不触发发送。
- `done` 后恢复输入框、建议项、表单交互。
- `error` / `abort` 后恢复交互，不残留永久锁定。
